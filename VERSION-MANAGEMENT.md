# HermesNexus 版本号管理规范

## 版本号策略

### 当前版本
- **项目版本**: v1.0.1 (MVP正式版本后的补丁版本)
- **发布日期**: 2026年4月11日
- **Git标签**: v1.0.1-mvp

### 版本号规则
遵循语义化版本号 (Semantic Versioning) 规范：`MAJOR.MINOR.PATCH`

- **MAJOR** (主版本号): 重大架构变更，不兼容的API修改
  - 示例: 1.0.0 → 2.0.0 (数据库架构重大变更)
  
- **MINOR** (次版本号): 向后兼容的功能新增
  - 示例: 1.0.0 → 1.1.0 (新增任务类型支持)
  
- **PATCH** (补丁版本号): 向后兼容的问题修复
  - 示例: 1.0.0 → 1.0.1 (bug修复)

## 版本更新流程

### 1. 确定版本变更类型
```bash
# 根据变更类型选择版本号更新
MAJOR_VERSION=1
MINOR_VERSION=0  
PATCH_VERSION=0
```

### 2. 更新版本号
```bash
# 更新VERSION.json
# 更新各组件版本号
# 更新文档中的版本号引用
```

### 3. 创建Git标签
```bash
git tag -a v1.0.0 -m "Release v1.0.0: MVP正式版本"
git push origin v1.0.0
```

## 版本兼容性

### API兼容性保证
- v1.x.x 版本保持向后兼容
- 破坏性变更必须在MAJOR版本号中体现
- 废弃功能需要保留至少一个MINOR版本

### 数据库Schema版本
- 数据库Schema版本独立管理
- 升级脚本必须向下兼容
- 变更记录在VERSION.json中
