# 54 测试与联调规范

## 目标

把 HermesNexus 的本机开发测试、开发服务器联调和 MVP 验证拆成两层：

1. 本机负责快速反馈。
2. 开发服务器负责真实部署验证。

## 测试分层

### 1. 本机单元测试

用途：快速验证代码逻辑。

覆盖内容：

- 任务状态机
- 配置解析
- 请求参数校验
- 数据模型
- Hermes 隔离逻辑

建议：

- 任何功能改动先跑本机单测
- 单测不依赖开发服务器
- 单测不依赖真实 Edge 节点

### 2. 本机轻量集成测试

用途：验证模块之间是否能对上。

覆盖内容：

- API 与数据库
- Cloud 服务核心流程
- 任务创建与状态查询
- 假 Edge / Mock Edge 的结果回写

建议：

- 优先用 mock 替代真实远端
- 用于开发阶段快速排错

### 3. 开发服务器联调测试

用途：验证真实部署路径。

覆盖内容：

- Cloud API 启动
- Edge Node 注册
- Heartbeat 心跳
- 任务拉取和执行
- 结果回写
- 审计和日志

建议：

- 开发服务器作为唯一联调环境
- 真实跑一遍“创建任务 → Edge 执行 → 状态回传”闭环
- 失败时保留日志和请求响应

### 4. MVP 验收测试

用途：判断这一轮是否可以进入下一阶段。

必须通过的验证项：

- 服务可启动
- 节点可注册
- 心跳可达
- 任务可下发
- 结果可回传
- 状态可查询
- 日志可追踪

## 测试原则

- 测试必须可重复
- 测试数据必须隔离
- 本机测试和服务器测试要分开
- 外部依赖尽量可模拟
- 关键闭环必须有端到端覆盖

## 推荐测试流程

### 本机开发时

1. 改代码
2. 跑单测
3. 跑本机轻量集成测试
4. 修复问题
5. 再跑一次

### 上开发服务器前

1. 确认本机测试通过
2. 同步代码到开发服务器
3. 启动 Cloud 和 Edge
4. 跑联调脚本
5. 记录结果

## 验证重点

- 状态是否一致
- 日志是否完整
- 失败是否能定位
- 重启后能否恢复
- 消息是否会重复执行

## 产出物

- 测试报告
- 联调问题列表
- 修复记录
- 回归结果
- MVP 验收结论
