# HermesNexus Phase 3 开发计划

Date: 2026-04-12
Version: 1.0.0

## 1. 阶段定位

Phase 3 的目标不是扩展功能边界，而是把 Phase 2 已完成的云控制平面推进到“可信、可控、可恢复”的安全与可靠性阶段。

本阶段重点解决三件事：
- 谁在操作、谁在执行：身份可信
- 什么能做、什么必须审批：权限可控
- 出问题怎么退回去：回滚可恢复

## 2. Phase 2 与 Phase 3 的关系

### Phase 2 已完成的内容
Phase 2 已完成基础 CI/CD、监控、告警、测试收口和部署文档收口，属于“可持续交付、可观测、可重复发布”的工程状态。

这意味着：
- CI/CD 基础流程已完成
- 需要做的是验证性部署，不是重新设计 CI/CD
- 下一阶段应转入安全与可靠性，不再继续扩展 Phase 2 的功能边界

### 当前建议
- 保留 Phase 2 的 CI/CD 作为基础门禁
- 进行一次实际部署 / 重启 / 回滚验证
- 在此基础上进入 Phase 3

## 3. Phase 3 的总体目标

1. 建立节点身份体系
2. 建立用户 / 节点 / 设备的认证与授权
3. 建立风险分级与审批流
4. 建立可验证的回滚与故障恢复机制
5. 建立完整的安全审计链路
6. 建立 Phase 3 的安全链路测试与验收标准

## 5. 范围定义

### 5.1 必须做
- 节点身份体系
- 节点认证 / token / mTLS 最小闭环
- 用户与角色授权
- 风险分级
- 审批流
- 回滚机制
- 故障恢复
- 安全审计链路

### 5.2 暂不做

## 6. 工作包拆分

### Workstream A：节点身份与信任基座

#### 目标
让系统先回答“谁是谁”。

#### 要做的事
- 为每个边缘节点分配唯一身份
- 定义节点注册、绑定、注销流程
- 定义设备身份与节点身份的关系
- 定义节点 token / 证书生命周期
- 定义云边通信认证入口

#### 交付物
- 节点身份模型
- 注册 / 绑定 / 注销流程说明
- 节点认证接口或服务定义
- 证书 / token 生命周期文档

#### 验收标准
- 每个节点可被唯一识别
- 未授权节点无法接入
- 凭证可轮换、可吊销

### Workstream B：权限与审批流

#### 目标
让系统先回答“谁能做什么”。

#### 要做的事
- 定义按租户、区域、设备类型、动作、风险等级的权限模型
- 定义低 / 中 / 高风险动作
- 定义审批状态机
- 定义审批通过 / 拒绝 / 超时 / 撤回流程
- 将审批结果写入审计链路

#### 交付物
- 权限模型说明
- 风险分级规则
- 审批流状态机
- 审批 API / 控制台交互草案
- 审批审计记录格式

#### 验收标准
- 高危动作不会直接执行
- 审批通过后才能下发
- 审批拒绝后任务无法继续
- 审批全过程可追踪

### Workstream C：回滚与故障恢复

#### 目标
让系统先回答“出问题怎么退回去”。

#### 要做的事
- 定义配置回滚、服务回滚、设备回滚、任务回滚
- 定义变更前快照
- 定义执行后验证
- 定义失败补偿和人工介入边界
- 定义节点离线、认证失败、配置错误、版本不兼容等恢复策略

#### 交付物
- 回滚策略说明
- 故障分类清单
- 恢复流程说明
- 回滚验证测试
- 故障演练清单

#### 验收标准
- 失败任务能进入回滚路径
- 回滚后可再次验证
- 关键任务有可逆方案
- 故障场景有明确处理方式

### Workstream D：安全审计与测试收口

#### 目标
让系统先回答“发生了什么，谁做的，能不能复盘”。

#### 要做的事
- 统一记录发起人、审批人、执行人、执行结果、是否回滚
- 每个 JobStep 单独记录
- 每个设备单独记录最终状态
- 保留失败证据
- 能从审计日志重建执行路径
- 补充安全链路的集成测试和 E2E 测试

#### 交付物
- 安全审计字段规范
- 安全事件日志格式
- 安全链路测试用例
- 故障演练报告模板

#### 验收标准
- 操作可追溯
- 审批可追溯
- 回滚可追溯
- 失败可定位

## 7. 推荐执行顺序

### Day 1：节点身份与认证边界

#### 目标
先把身份层打通。

#### 任务
- 梳理节点、设备、用户、自动化 Agent 的身份边界
- 定义节点注册 / 绑定 / 注销流程
- 定义节点 token / 证书的最小方案
- 梳理云边通信的认证入口

#### 交付物
- 身份边界说明
- 节点身份流程图 / 文档
- 节点认证最小设计

#### 验收
- 节点身份概念清晰
- 能明确哪些请求需要身份校验

### Day 2：授权模型与风险分级

#### 目标
先把权限边界定住。

#### 任务
- 定义租户 / 区域 / 设备类型 / 动作级权限
- 设计风险分级规则
- 定义默认拒绝策略
- 定义高危动作白名单 / 黑名单边界

#### 交付物
- 权限矩阵
- 风险分级表
- 授权策略说明

#### 验收
- 能判断一个动作是否允许执行
- 能判断一个动作是否需要审批

### Day 3：审批流最小闭环

#### 目标
把高危任务从“创建”到“批准”串起来。

#### 任务
- 定义任务草案 / 审批中 / 已批准 / 已拒绝 / 已撤回状态
- 设计审批 API / 数据结构
- 将审批记录写入审计链路
- 处理超时与撤回

#### 交付物
- 审批流状态机
- 审批记录模型
- 审批测试用例

#### 验收
- 高危任务必须审批后才能执行
- 审批结果可查询

### Day 4：回滚与故障恢复

#### 目标
让任务和变更可逆。

#### 任务
- 为任务定义回滚计划
- 定义配置 / 服务 / 设备回滚路径
- 定义失败场景处理规则
- 定义恢复后的再次验证步骤

#### 交付物
- 回滚流程说明
- 故障恢复清单
- 回滚演练用例

#### 验收
- 至少核心任务类型具备回滚路径
- 故障后可恢复到可验证状态

### Day 5：安全审计与验收测试

#### 目标
把 Phase 3 的安全链路收口成可验收版本。

#### 任务
- 统一审计字段
- 补充关键安全链路测试
- 补充故障恢复测试
- 更新部署与验收文档
- 输出 Phase 4 候选优先级

#### 交付物
- 安全审计规范
- 安全链路测试结果
- Phase 3 完成总结
- Phase 4 规划草案

#### 验收
- 安全链路可重复验证
- 文档与实现一致
- 下一阶段优先级清晰

## 9. Phase 3 完成标准

满足以下条件即可认为 Phase 3 达标：
- 节点身份明确
- 认证与授权可用
- 高危动作必须审批
- 回滚机制可执行
- 故障恢复可验证
- 审计链路完整
- 安全测试可重复执行

## 10. 风险与缓解

### 风险 1：安全能力一次性做得过重
缓解：先做最小闭环，优先 token / 身份 / 审批 / 回滚，不要一开始就做完整 IAM 平台。

### 风险 2：审批流与任务模型分离
缓解：审批必须是任务生命周期的一部分，而不是独立旁路系统。

### 风险 3：回滚只停留在文档
缓解：至少为关键任务类型做可执行回滚测试。

### 风险 4：mTLS / KMS / Vault 方案过早上量
缓解：先把可信边界和短期凭证做稳，再逐步增强密钥体系。

## 11. 对 CI/CD 和部署的结论

- Phase 2 的 CI/CD 已完成基础版本
- 现在不需要重新建设 CI/CD，而是用它做门禁
- 需要做的是：部署验证、重启验证、回滚验证
- Phase 3 可以在现有 CI/CD 上继续推进，不必等待更重的 DevOps 平台
