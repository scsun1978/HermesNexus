# HermesNexus Phase 2 Week 2 开发计划

Date: 2026-04-12
Version: 1.0.0

## 目标

在 Week 1 已完成云控制平面骨架的基础上，推进 Phase 2 Full 的第一段工作：把系统从“内存可跑”推进到“可持久化、可鉴权、可回归、可验证”。

- `docs/68-技术栈决策与MVP边界.md`

## 当前结论

Week 1 已完成对象模型、参数化部署、资产管理、任务编排、审计记录和控制台骨架。

Week 2 不再做“功能骨架”，而是补齐 Phase 2 Full 的关键短板：
- 数据持久化
- API 安全
- 自动化测试
- 部署与验证稳定性

## Week 2 范围

### 必做项
1. 数据持久化落地
2. API 认证与最小权限控制
3. 核心流程自动化测试
4. 部署与验证脚本收敛
5. 文档与验收标准更新

### 暂缓项
- 多协议接入
- 高可用集群
- 多机房部署
- 复杂审批流
- 大规模调度优化
- Hermes 深度集成

## 推荐执行顺序

### Day 1: 数据持久化方案冻结

**目标：** 让资产、任务、审计这三条主线有明确的持久化目标。

**任务：**
- 确认当前内存存储覆盖范围
- 定义 SQLite 作为 Week 2 首选持久化后端
- 列出需要落盘的对象：assets / tasks / audit_logs / nodes
- 明确主键、索引、更新时间、关联字段
- 评估现有 API 与数据层的最小改造面

**交付物：**
- 持久化方案说明
- 数据模型映射表
- 迁移范围清单

**验收：**
- 能明确哪些数据必须持久化
- 能明确哪些字段必须保留

### Day 2: 实现最小持久化存储

**目标：** 让核心数据不再依赖进程内存。

**任务：**
- 实现 SQLite 数据库连接与初始化
- 实现资产持久化读写
- 实现任务持久化读写
- 实现审计日志持久化读写
- 保留最小兼容层，避免一次性重写所有 API

**交付物：**
- 持久化存储实现
- 初始化脚本或迁移脚本
- 最小数据访问层

**验收：**
- 重启后数据可恢复
- 资产、任务、审计至少各有一个持久化路径

### Day 3: API 安全最小化落地

**目标：** 给控制平面加上基础访问保护。

**任务：**
- 定义认证入口（例如 token / header / 简化 JWT 方案）
- 为高风险接口增加认证校验
- 为写操作增加最小权限检查
- 为失败认证增加统一错误响应
- 维持开发环境可调试性

**交付物：**
- 认证中间件或依赖注入方案
- 权限规则草案
- 安全配置项

**验收：**
- 未认证请求被拒绝
- 关键写接口受保护
- 开发环境可通过配置关闭或切换认证模式

### Day 4: 关键接口回归与测试补齐

**目标：** 把 Week 2 的改动用测试锁住。

**任务：**
- 为持久化层补测试
- 为认证层补测试
- 为资产 / 任务 / 审计的核心 CRUD 补测试
- 为重启恢复补回归测试
- 为错误路径补测试

**交付物：**
- 单元测试
- 集成测试
- 测试数据夹具

**验收：**
- 核心路径测试可自动执行
- 失败时能定位到具体模块

### Day 5: 部署脚本与配置收敛

**目标：** 让 Week 2 的持久化与安全配置可以稳定落地。

**任务：**
- 更新环境变量说明
- 更新数据库配置说明
- 更新启动脚本参数
- 校验本机 / 开发服务器配置一致性
- 清理 Week 1 遗留的临时配置

**交付物：**
- 更新后的配置规范
- 更新后的启动脚本
- 配置验证脚本更新

**验收：**
- 本机与开发服务器使用一致的启动逻辑
- 持久化配置可被验证

### Day 6: smoke / e2e 补强

**目标：** 用端到端流程验证 Week 2 的改动。

**任务：**
- 更新 smoke 测试脚本
- 增加持久化恢复验证
- 增加认证失败验证
- 增加关键接口的 e2e 回归
- 补充运行说明

**交付物：**
- smoke 流程更新
- e2e 场景更新
- 验证报告草案

**验收：**
- smoke 能覆盖持久化与认证
- e2e 能覆盖重启恢复和关键写接口

### Day 7: 文档收口与阶段确认

**目标：** 把 Week 2 做成可交接、可复用、可继续推进的状态。

**任务：**
- 更新 Phase 2 Full 说明
- 更新部署和验收文档
- 整理本周问题与已知限制
- 输出下一步 Phase 2 Full 后半段优先级

**交付物：**
- Week 2 完成总结
- 更新后的验收清单
- 下一阶段计划草案

**验收：**
- 文档与实际实现一致
- 下一周任务清晰

## Week 2 完成标准

满足以下条件即可认为 Week 2 达标：
- 核心数据可持久化
- 关键接口有基础安全保护
- 核心路径有自动化测试
- 配置和部署脚本稳定
- smoke / e2e 能验证 Week 2 改动
- 文档可执行

## Week 2 风险

1. **持久化改造面大**
   - 缓解：先做最小可用 SQLite，再逐步抽象

2. **安全改造影响接口兼容性**
   - 缓解：先保护写接口，再逐步扩展

3. **测试补齐耗时**
   - 缓解：先锁住最关键的资产 / 任务 / 审计主路径

4. **文档与实现不同步**
   - 缓解：每完成一个子任务就同步更新说明

## 推荐的 Week 2 现实优先级

1. 数据持久化
2. API 安全
3. 测试补齐
4. 配置收敛
5. smoke / e2e
6. 文档收口

## 备注

Week 2 的定位是 Phase 2 Full 的开端，不是 Phase 3。

如果 Week 2 完成得好，下一步就可以进入：
- 更完整的监控告警
- 更严格的权限和认证
- 更系统的部署验证
- Phase 3 的安全与可靠性增强
