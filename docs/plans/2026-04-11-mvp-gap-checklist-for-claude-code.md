# HermesNexus 完整 MVP 整改清单（给 Claude Code）

Date: 2026-04-11

## 目标

把当前“简化版 demo 已部署”的状态，推进到“完整 MVP 可验收”的状态。

当前已确认的事实：
- 云端与边缘节点都已在开发服务器上跑起来
- 健康检查和基础统计接口可用
- 边缘节点可以注册、发心跳、执行任务
- 但任务结果没有回写到云端控制平面
- 当前远端部署的是简化版服务，不是完整项目 API

## 结论

完整 MVP 目前未完成。最大阻塞项是“任务结果回写闭环”以及“完整 API 部署切换”。

---

## 一、必须修复项（阻塞完整 MVP）

### 1. 任务结果必须从 Edge 回写到 Cloud

现状：
- edge 节点本地执行任务后，只记日志
- 云端 job 状态仍是 pending

要求：
- edge 执行完成后，调用云端结果回传接口
- 云端收到结果后，更新 job 状态为 completed / failed
- 同时写入 event 和 audit log

验收标准：
- 创建一个任务后，edge 执行完成
- 云端 job 状态从 pending 变为 completed
- 云端可查询到任务结果
- events / audit_logs 中可看到对应记录

建议检查点：
- `simple-edge-node.py`
- `cloud/api/main.py` 的 `/api/v1/nodes/{node_id}/tasks/{task_id}/result`
- 任务状态流转是否完整

---

### 2. 部署目标切换到完整项目 API

现状：
- 开发服务器上运行的是 `simple-cloud-api.py`
- 这不是完整项目的 `cloud/api/main.py`

要求：
- 用完整 API 启动控制平面
- 验证以下完整接口链路：
  - `/api/v1/nodes`
  - `/api/v1/nodes/{node_id}/register`
  - `/api/v1/nodes/{node_id}/heartbeat`
  - `/api/v1/nodes/{node_id}/tasks`
  - `/api/v1/nodes/{node_id}/tasks/{task_id}/result`
  - `/api/v1/devices`
  - `/api/v1/jobs`
  - `/api/v1/jobs/{job_id}`
  - `/api/v1/events`
  - `/api/v1/stats`

验收标准：
- 不再依赖简化 demo 才能完成主要闭环
- 完整 API 直接支持 MVP 验收流程

---

### 3. 任务状态机要完整落地

现状：
- 云端可以创建任务
- edge 可以执行任务
- 但状态机没有完全闭环

要求：
- 明确任务状态流转：pending -> running -> completed/failed
- 结果回写时，云端状态同步更新
- 节点拉取任务时，状态要合理切换

验收标准：
- 任务创建后初始为 pending
- 节点开始执行时状态进入 running
- 执行完成后状态进入 completed / failed
- 查询接口返回的状态与日志一致

---

### 4. 重启恢复验证

现状：
- 还没有完整验证服务重启后的恢复行为

要求：
- 重启 cloud 后，节点能重新连上
- 重启 edge 后，能自动恢复注册和心跳
- 数据库中的 job/device/event/audit 不丢失

验收标准：
- 重启后服务自动恢复到可用状态
- 任务/节点状态不乱
- 历史记录仍可查

---

## 二、建议修复项（影响 MVP 质量）

### 5. 提供一键启动/停止脚本

建议：
- `scripts/start-cloud.sh`
- `scripts/start-edge.sh`
- `scripts/stop-cloud.sh`
- `scripts/stop-edge.sh`
- 或者统一的 `systemd` / `compose` 管理方式

目标：
- 避免只靠 nohup 手工启动
- 保证每次部署方式一致

---

### 6. 补齐设备管理完整流程

要求：
- 创建设备
- 查询设备
- 设备绑定任务
- 设备与任务关联关系在 API 返回中可见

验收标准：
- /api/v1/devices 可用
- 设备创建后，job 创建能正确关联设备

---

### 7. 事件与审计链路要完整可查

要求：
- 任务创建事件
- 任务完成事件
- 任务失败事件
- 取消任务事件
- 节点错误事件
- 审计日志可查

验收标准：
- 事件列表能看到完整闭环
- 审计记录能定位任务和节点操作

---

### 8. 把真实部署环境纳入自动化回归

要求：
- E2E 测试覆盖真实 dev server
- 不只是本地单测通过
- 至少包含：注册、心跳、任务下发、执行、回写、查询

验收标准：
- 一条 E2E 测试能稳定跑通完整闭环
- 改代码后可重复回归

---

1. 先修任务结果回写
2. 再切换到完整项目 API
3. 再验证任务状态机
4. 再做重启恢复
5. 再补脚本和自动化回归
6. 最后补监控与安全增强

---

## 五、完整 MVP 验收标准

只有满足以下条件，才算完整 MVP：

- 本机可稳定开发和跑测试
- 开发服务器运行的是完整项目 API
- edge 节点能稳定注册和发心跳
- 任务能从云端下发到节点
- 节点执行结果能回写云端
- 云端状态、事件、审计记录一致
- 重启后系统能恢复
- 有可重复执行的部署与回归流程

---
