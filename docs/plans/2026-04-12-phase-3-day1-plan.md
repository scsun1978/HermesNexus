## 🎯 Phase 3 Day 1 执行计划

### Workstream A：节点身份与信任基座

#### 任务1: 身份边界梳理 🟡
**目标**: 明确系统中的各类身份主体和边界

**身份主体分类**:
1. **用户** - 操作系统的操作者
   - 管理员、操作员、只读用户
   - 通过API Key或Token认证

2. **边缘节点** - 执行任务的物理/virtual机器
   - 唯一节点ID
   - 通过节点Token或证书认证

3. **设备** - 被管理的目标资产
   - 设备ID和元数据
   - 属于某个节点管理

4. **自动化Agent** - 系统内部自动化流程
   - 系统用户身份
   - 具有特定权限范围

**验收标准**:
- [ ] 各类身份主体的定义清晰
- [ ] 身份之间的边界关系明确
- [ ] 认证流程图完成

#### 任务2: 节点注册/绑定/注销流程 🟡
**目标**: 定义节点的完整生命周期管理

**节点生命周期状态**:
```
UNREGISTERED → REGISTERING → REGISTERED → ACTIVE → INACTIVE → DEREGISTERED
                ↓            ↓          ↓         ↓
             FAILED       EXPIRED    SUSPENDED  DECOMMISSIONED
```

**关键流程**:
1. **注册流程**: 节点首次接入，提交身份信息
2. **绑定流程**: 节点与设备关联，建立信任关系
3. **心跳维持**: 定期更新状态，保持活跃
4. **注销流程**: 主动或被动下线，清理身份

**验收标准**:
- [ ] 节点生命周期状态机定义
- [ ] 注册流程API设计
- [ ] 绑定关系数据模型
- [ ] 注销清理机制

#### 任务3: 节点Token/证书最小方案 🟡
**目标**: 实现最小可行的节点认证机制

**技术方案对比**:
| 方案 | 复杂度 | 安全性 | 实施时间 | 推荐度 |
|------|--------|--------|----------|--------|
| 简单Token | 低 | 中 | 1天 | ⭐⭐⭐ |
| JWT + 签名 | 中 | 中高 | 2天 | ⭐⭐⭐⭐ |
| mTLS | 高 | 高 | 5天+ | ⭐ |

**最小方案选择**: JWT + 节点证书签名
- 节点注册时颁发JWT Token
- Token包含节点ID、权限、过期时间
- 云边通信时验证Token签名

**验收标准**:
- [ ] 节点认证Token设计
- [ ] Token颁发和验证API
- [ ] Token刷新和吊销机制
- [ ] 认证中间件集成

#### 任务4: 云边通信认证入口 🟡
**目标**: 在现有云边通信中添加认证检查

**当前云边通信点**:
1. **节点注册**: `/api/v1/nodes/{node_id}/register`
2. **心跳上报**: `/api/v1/nodes/{node_id}/heartbeat`
3. **任务获取**: `/api/v1/nodes/{node_id}/tasks`
4. **结果上报**: `/api/v1/nodes/{node_id}/tasks/{task_id}/result`

**认证集成点**:
- 在每个端点添加Token验证
- 区分不同操作类型的权限要求
- 记录认证失败事件

**验收标准**:
- [ ] 云边通信认证中间件
- [ ] 分级权限检查
- [ ] 认证失败处理
- [ ] 审计日志记录
