# Hermes 分布式边缘设备管理系统 下一步落地步骤

Date: 2026-04-12
Version: 1.0.2

## 现状判断

当前不是“从零开发阶段”，而是“核心闭环已实现，进入补缺口与验收阶段”。

通过 graphify-out 分析和当前源码可确认：
- 云端控制面已具备节点注册、心跳、任务查询/下发、结果回写、审计/事件记录能力
- 边缘执行面已具备自动注册、心跳循环、任务轮询、SSH 任务执行、结果回传能力
- 共享层已经有协议、错误码、模式数据结构和数据库/DAO 支撑

仍然存在的主要缺口：
- 脚本类任务执行未完成
- 部分实现需要清理/统一（协议命名、状态流转、返回字段）
- 需要把核心链路做成稳定的可重复验证流程
- 需要把文档从“规划中”同步成“现状 + 剩余工作”

## 目标

下一步不再追求“把核心能力再做一遍”，而是完成下面四件事：
1. 验证当前核心闭环是否真的可用
2. 补齐明确缺口
3. 把部署与测试流程固定下来
4. 把文档更新为与当前代码状态一致

## 近期原则

1. 先验证，后新增。
2. 只补明确缺口，不提前扩展到 HA、多机房、复杂权限。
3. 本机负责开发与单测，开发服务器负责联调与端到端验证。
4. 任何新功能都必须落到测试和验收步骤，不允许只写规划。

### 已完成
- 节点注册
- 心跳上报
- 任务下发与轮询
- 结果回传
- 审计记录
- 共享消息模型

### 部分完成

- 任务失败、重复回传、超时回传是否完全不污染状态

## 下一步落地顺序

### Step 1: 先做代码状态复核

**目标：** 先确认哪些是已完成、哪些是缺口，避免重复开发。

**要核对的内容：**
- 云端节点注册与心跳链路
- 边缘节点自动注册与心跳循环
- 任务查询/执行/回传链路
- 脚本任务是否仍然缺失
- 文档中描述的“未来目标”是否已经变成“当前实现”

**产出：**
- 一份当前实现清单
- 一份缺口清单
- 一份优先级排序

### Step 2: 补齐最明确的功能缺口

**目标：** 只补最明确、最影响闭环完整性的缺口。

**优先补的内容：**
- 脚本类任务执行
- 任务状态流转一致性
- 断线/重连异常路径
- 回传重复数据的幂等处理

**验收标准：**
- 核心闭环不依赖人工干预
- 失败路径不会污染任务状态
- 同一任务重复回传不会把状态改乱

### Step 3: 固化验证流程

**目标：** 把“代码看起来有了”变成“每次都能复现地验证通过”。

**要做的事：**
- 本机单测补齐
- 开发服务器联调脚本固定
- 集成测试脚本固定
- 失败场景脚本固定

**验收标准：**
- 本机能跑单元测试
- 开发服务器能跑集成测试
- 失败时能快速定位到具体步骤

### Step 4: 同步部署与运维文档

**目标：** 让后续调试、交接和回归不依赖口头说明。

**文档内容：**
- 当前实现范围
- 当前已知缺口
- 本机如何跑
- 开发服务器如何联调
- 节点如何启动和重连
- 常见故障如何排查

## 建议的本轮交付物

1. 当前实现清单
2. 缺口清单
3. 脚本任务执行补齐
4. 核心链路验证脚本
5. 开发服务器联调脚本
6. 运维排障文档

## MVP 判断线

如果做到下面这些，就可以认为下一步没有跑偏：
- 核心链路可重复验证
- 脚本任务缺口已关闭
- 断线重连和恢复逻辑可用
- 失败/重复回传不会污染状态
- 开发服务器与本机结论一致

## 不做项

本轮先不做：
- 多机房
- 高可用集群
- 复杂权限体系
- 大规模调度优化
- 完整监控平台

## 推荐推进方式

按顺序推进：
1. 代码状态复核
2. 缺口补齐
3. 验证流程固化
4. 部署与运维文档同步
