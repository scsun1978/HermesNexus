# Hermes 分布式边缘设备管理系统 下一步落地步骤

Date: 2026-04-12
Version: 1.0.1

## 目标

在不偏离主线的前提下，先做出一个最小可验证闭环：设备注册 → 心跳 → 任务下发 → 结果回传 → 断线重连。把“能部署”推进到“能管设备”。

## 近期原则

1. 本机只负责开发、单测和快速调试。
2. 开发服务器负责集成验证和端到端联调。
3. 先用容器或虚拟机模拟边缘节点，后续再替换真实设备。
4. 先做最小闭环，不做大规模调度、不做复杂权限、不做高可用集群。

## 下一步落地顺序

### Step 1: 冻结最小控制面协议

**目标：** 把服务端和边缘节点之间最少要说清楚的话先定下来。

**要定义的内容：**
- 节点注册请求与响应
- 心跳消息
- 任务下发消息
- 任务结果回传消息
- 节点离线 / 重连消息
- 消息里必须有的字段：node_id、task_id、timestamp、status、payload、error

**产出：**
- 一份协议说明文档
- 一份最小 JSON 示例
- 一份错误码约定

### Step 2: 建设备注册与心跳最小服务

**目标：** 让服务端能识别节点是否在线。

**服务端能力：**
- 接收节点注册
- 生成或确认 node_id
- 维护节点最后心跳时间
- 标记在线 / 离线状态

**边缘节点能力：**
- 启动后自动注册
- 定时发心跳
- 断线后自动重连并重新注册

**验收标准：**
- 服务端能看到节点列表
- 节点掉线后状态能变化
- 节点重启后能恢复在线

### Step 3: 打通任务下发与结果回传

**目标：** 让服务端真正能“管”设备，而不是只看设备在线。

**服务端能力：**
- 创建任务
- 下发任务到指定节点
- 跟踪任务状态

**边缘节点能力：**
- 接收任务
- 执行任务
- 回传结果和失败原因

**验收标准：**
- 服务端发一个测试任务，节点能收到
- 节点执行后能回传结果
- 服务端能记录任务成功或失败

### Step 4: 建立开发服务器部署方式

**目标：** 把上面的最小链路放到可重复部署的环境里。

**要做的事：**
- 准备服务端启动方式
- 准备边缘节点模拟器启动方式
- 固定配置文件位置
- 固定日志目录
- 固定数据目录

**验收标准：**
- 服务器上可以一键启动服务端和模拟节点
- 重启后不需要手工修很多状态
- 日志可以直接定位问题

### Step 5: 建立最小集成测试

**目标：** 每次改动都能自动验证核心链路。

**测试项：**
- 服务端启动测试
- 节点注册测试
- 心跳测试
- 任务下发测试
- 结果回传测试
- 断线重连测试

**验收标准：**
- 本机能跑单元测试
- 开发服务器能跑集成测试
- 失败时能快速定位到具体步骤

### Step 6: 补最小运维和排障文档

**目标：** 让后续调试和交接成本降低。

**文档内容：**
- 本机开发怎么跑
- 开发服务器怎么部署
- 模拟节点怎么启动
- 常见故障怎么排查
- 如何重置环境

## 建议的本轮交付物

1. 协议说明文档
2. 服务端注册/心跳接口
3. 边缘节点模拟器
4. 任务下发与结果回传的最小闭环
5. 开发服务器部署脚本
6. 集成测试脚本
7. 运维排障文档

## MVP 判断线

如果做到下面这些，就可以认为下一步没有跑偏：
- 本机可开发、可单测
- 开发服务器可部署、可联调
- 节点可注册、可心跳
- 服务端可下发任务
- 节点可回传结果
- 重启后可恢复

## 不做项

本轮先不做：
- 多机房
- 高可用集群
- 复杂权限体系
- 大规模调度优化
- 完整监控平台

## 推荐推进方式

按顺序推进：
1. 协议
2. 注册/心跳
3. 任务闭环
4. 服务器部署
5. 集成测试
6. 文档
