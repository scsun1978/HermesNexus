# HermesNexus Phase 2 Week 3 开发计划

Date: 2026-04-12
Version: 1.0.0

## 目标

Week 3 的定位不是进入 Phase 3，而是对 Phase 2 Full 进行收口：把系统从“已可用”推进到“可稳定验证、可重复部署、可持续迭代”。

本周重点：
- 测试完善
- E2E / smoke 固化
- 性能基线与基础优化
- 部署与文档收口

## 规划依据

- `docs/41-实施里程碑与交付拆分.md`
- `docs/09-MVP开发路线图.md`
- `docs/plans/2026-04-12-phase-2-cloud-control-plane-plan.md`
- `docs/plans/2026-04-12-phase-2-week2-project-summary.md`
- 当前 Week 1 / Week 2 已完成代码与测试结果

## 当前结论

Week 1 已完成云控制平面骨架，Week 2 已完成数据持久化、API 安全与测试框架。

Week 3 不再重复证明“能跑通”，而是要把现有能力变成稳定的工程交付能力：
- 关键链路可重复验证
- 故障能快速定位
- 部署流程可复现
- 文档与实际实现保持一致

## Week 3 范围

### 需要做的
- 补充集成测试和回归测试
- 补充 / 固化 E2E 与 smoke 流程
- 为数据库/API 路径建立性能基线
- 对明显慢路径做第一轮优化
- 收敛启动、配置、部署和排障文档
- 整理 Week 4 的后续优先级

### 暂不做的
- 多协议接入扩展
- 高可用集群
- 多机房部署
- 复杂审批流
- 大规模调度优化
- Phase 3 节点身份/更重安全体系

## 推荐推进顺序

### Day 1：测试基线整理

**目标：** 统一当前验收依据，清理测试边界。

**任务：**
- 梳理当前可执行测试范围
- 区分单元测试、集成测试、E2E、smoke、旧验证脚本
- 标注过时测试脚本，避免混入正式验收
- 建立 Week 3 测试执行清单

**交付物：**
- 测试基线说明
- 测试分类清单
- 旧脚本处理建议

**验收：**
- 能明确哪些测试可作为当前版本验收依据
- 旧脚本不再干扰正式验证

### Day 2：集成测试补强

**目标：** 锁住资产、任务、审计、认证和云边链路的核心流程。

**任务：**
- 补资产/任务/审计主线集成测试
- 补认证链路集成测试
- 补云边注册、心跳、任务下发、结果回传的关键测试
- 强化测试数据隔离和清理机制

**交付物：**
- 新增/更新的集成测试
- 测试夹具或隔离方案

**验收：**
- 核心链路有可重复执行的集成测试
- 不依赖手工环境污染

### Day 3：E2E / smoke 固化

**目标：** 把端到端验证做成稳定门禁。

**任务：**
- 整理现有 E2E 场景
- 固化 smoke 检查流程
- 增加失败时的定位输出
- 统一启动、检查、清理步骤

**交付物：**
- smoke 执行脚本
- E2E 执行脚本
- 运行说明

**验收：**
- 一条命令可跑 smoke
- 一条命令可跑 E2E
- 失败时能快速定位问题

### Day 4：性能基线与基础优化

**目标：** 找出当前最慢路径，为后续优化提供依据。

**任务：**
- 为数据库查询建立基线
- 识别最慢的 DAO / API 路径
- 检查索引是否足够
- 评估是否需要连接池或查询优化

**交付物：**
- 性能基线记录
- 优化建议清单
- 第一轮性能修复（如有必要）

**验收：**
- 有可参考的性能基线
- 明确后续优化优先级

### Day 5：部署与文档收口

**目标：** 把 Week 3 变成可交接、可复现、可继续推进的状态。

**任务：**
- 更新部署说明
- 更新配置模板和启动入口说明
- 更新验收清单
- 补充故障排障 Runbook
- 输出 Week 4 优先级建议

**交付物：**
- Week 3 完成总结
- 更新后的部署/验证文档
- 下一阶段计划草案

**验收：**
- 文档与代码一致
- 下一周可以直接接续推进

## Week 3 完成标准

满足以下条件即可认为 Week 3 达标：
- 核心测试链路清晰可执行
- 集成测试与 E2E / smoke 可重复运行
- 关键性能路径有基线记录
- 部署与排障文档收口完成
- Week 4 优先级明确

## Week 3 风险

1. **测试边界混乱**
   - 缓解：明确旧脚本与正式验收脚本的边界

2. **集成测试依赖环境不稳定**
   - 缓解：优先做隔离和清理机制

3. **性能优化范围过大**
   - 缓解：只优化当前最慢的关键路径，不扩散到平台级重构

4. **文档与实现不同步**
   - 缓解：每天交付后同步更新说明

## Week 3 建议优先级

1. 测试基线整理
2. 集成测试补强
3. E2E / smoke 固化
4. 性能基线与基础优化
5. 部署与文档收口

## Week 3 定位

Week 3 是 Phase 2 Full 的收口周，不是 Phase 3 的开始。

如果 Week 3 完成得好，下一阶段再进入：
- 安全与可靠性增强
- 节点身份与认证
- 更严格的回滚机制
- 多协议设备接入
