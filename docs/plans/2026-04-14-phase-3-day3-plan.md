## 3. 核心概念设计

### 3.1 审批状态机

```
┌─────────────┐
│   DRAFT     │  草案状态
└──────┬──────┘
       │ submit()
       ▼
┌─────────────┐
│ PENDING     │  审批中
└──────┬──────┘
       │ approve()
       ▼        │ reject()
       │         ▼
┌─────────────┐ ┌─────────────┐
│ APPROVED    │ │ REJECTED    │
└─────────────┘ └─────────────┘
                    │ withdraw()
                    ▼
             ┌─────────────┐
             │ WITHDRAWN   │
             └─────────────┘

所有状态 ───> timeout() ───> EXPIRED
```

### 3.2 审批流程设计

```
1. 用户发起高风险操作
   ↓
2. 系统评估风险等级
   ↓
3. 判断是否需要审批
   ↓ (是)
4. 创建审批请求 (DRAFT → PENDING)
   ↓
5. 通知审批人
   ↓
6. 审批人审批 (APPROVED/REJECTED)
   ↓
7. 根据审批结果执行/拒绝操作
   ↓
8. 记录审计日志
```

### 风险1: 审批流程过于复杂
**缓解**: 先实现最小闭环，单人审批，后续可扩展多级审批

### 风险2: 审批性能问题
**缓解**: 异步处理，消息队列解耦

### 风险3: 审计记录不完整
**缓解**: 强制审计，关键操作必记录
