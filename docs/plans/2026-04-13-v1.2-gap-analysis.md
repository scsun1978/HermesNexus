# HermesNexus 现状 vs 下一阶段差距分析

Date: 2026-04-13  
Version: 1.0.0  
Status: 规划中

## 结论先说

当前仓库已经具备：
- 节点注册 / 心跳 / 结果回传链路
- 资产 / 任务 / 审计的单资源管理
- DAO 层批量读写能力
- 控制台基础页面
- 多套部署与验证脚本

下一阶段真正缺的，不是“能不能跑”，而是：
- 批量操作是否上到 API 层和 UI 层
- 多节点是否有更强的管理视图
- 部署流程是否统一参数化
- Smoke / E2E 是否收口成正式门禁

## 重点缺口拆解

### 1. 批量操作还停在 DAO / 测试层

现状：
- DAO 层已有 `select_by_ids()`、`insert_batch()`、`update_batch()`
- 性能测试已经覆盖批量写入 / 查询
- 但 API 层没有统一的批量资产 / 批量任务写入接口
- 控制台也没有批量选择 / 批量按钮

要补的块：
- 资产批量创建 / 更新 / 停用 API
- 任务批量创建 / 批量分发 API
- 前端多选和批量操作入口
- 批量结果结构与失败项拆分

### 2. 多节点管理可用，但还不够“可管理”

现状：
- 节点注册、心跳、任务拉取、结果回写已经跑通
- API 层可以列出节点、查看节点详情
- 但没有更强的节点管理页面能力

要补的块：
- 节点筛选 / 分组 / 批量查看
- 节点详情页补齐最近任务、审计摘要、状态变化
- 节点健康和在线状态的更清晰表达

### 3. 部署和验证流程还不够统一

现状：
- 有多套脚本：本地演示、开发服务器、部署引导、快速验证
- 说明已经有一部分流程化能力
- 但入口分散，复用性一般

要补的块：
- 统一启动 / 停止 / 验证入口
- 参数化 host / port / 目录 / 环境变量
- 标准化 smoke 输出

### 4. Smoke / E2E 还没有成为正式门禁

现状：
- 有 e2e、性能、快速验证脚本
- 但还没整理成“每次改动都照这个顺序跑”的正式门禁

要补的块：
- 固定顺序的 smoke 流程
- 失败时的定位提示
- 开发服务器侧复现步骤

4. 批量结果与失败摘要

1. 多节点管理增强

3. 部署流程参数化
