## 当前结论

MVP v1.0 已完成“最小闭环”的验证，下一阶段不再重复证明“能跑通”，而是进入 Phase 2：云端控制平面建设。

这一阶段的核心目标是把系统从“单链路可执行”推进到“可管理、可编排、可审计、可视化”的控制面。

## Phase 2 目标

构建云端控制平面最小可用版本，让系统具备以下能力：
- 资产管理
- 任务编排
- 审计记录
- 基础控制台
- 更稳定的契约与部署流程

## Phase 2 范围

### 需要做的
- 统一任务契约，收敛 jobs/tasks/devices/nodes 的边界
- 参数化部署脚本，减少固定路径依赖
- 将 smoke / e2e 验证整理成正式流程
- 建立资产模型与设备视图
- 建立任务编排接口
- 建立审计记录链路
- 建立最小控制台页面
- 提升本机和开发服务器之间的部署一致性

### 暂不做的
- 多协议接入扩展
- 高可用集群
- 多机房部署
- 复杂审批流
- 大规模调度优化
- 完整企业级权限体系

## 推荐推进顺序

### Task 1: 冻结云端控制平面对象模型

**目标：** 统一系统内对象定义，避免概念漂移。

**产出：**
- 设备模型
- 节点模型
- 任务模型
- 审计事件模型
- 状态枚举与错误码

**验收：**
- 文档中不再混用 jobs/tasks/devices/nodes
- 每类对象有清晰职责边界

### Task 2: 参数化部署与运行配置

**目标：** 让本机和开发服务器都能使用一致的启动方式。

**产出：**
- 环境变量规范
- 配置文件模板
- 部署脚本参数化
- 启停脚本统一入口

**验收：**
- 不依赖固定路径
- 本机和服务器都可复现启动

### Task 3: 建立资产管理最小能力

**目标：** 让控制平面能看见“管什么”。

**产出：**
- 资产登记接口
- 资产列表接口
- 资产详情接口
- 资产状态视图

**验收：**
- 能创建、查看、更新、查询资产
- 资产和边缘节点之间关系清晰

### Task 4: 建立任务编排最小能力

**目标：** 让控制平面能组织任务，而不是只发单条命令。

**产出：**
- 任务创建接口
- 任务分派接口
- 任务状态跟踪
- 任务结果归档

**验收：**
- 能创建任务并分配给目标节点
- 能查看任务状态变化

### Task 5: 建立审计记录链路

**目标：** 让所有关键动作可追踪、可回放。

**产出：**
- 审计记录表
- 审计写入逻辑
- 审计查询接口
- 基础审计视图

**验收：**
- 任务创建、分配、执行、回传都有记录
- 异常可追溯到节点和任务

### Task 6: 建立基础控制台

**目标：** 让系统状态可视化，而不是只靠日志和接口。

**产出：**
- 资产页
- 任务页
- 审计页
- 节点状态页

**验收：**
- 可查看关键状态
- 可定位任务执行结果

### Task 7: 完成 smoke / e2e 正式流程

**目标：** 让下一阶段有可重复的质量门禁。

**产出：**
- 启动 smoke 流程
- 任务闭环 e2e 流程
- 异常重连验证
- 部署后检查脚本

**验收：**
- 每次改动可以按固定脚本验证
- 失败时能快速定位问题

### Task 8: 补齐部署文档与验收标准

**目标：** 让别人可以照文档复现下一阶段能力。

**产出：**
- 本机开发指南更新
- 开发服务器部署指南更新
- Phase 2 验收清单
- 故障排障 Runbook

**验收：**
- 文档可执行
- 验收标准可检查

## Phase 2 完成标准

以下条件全部满足时，认为 Phase 2 可以收口：
- 资产、任务、审计三条主线可用
- 控制台能展示核心状态
- 部署脚本参数化完成
- smoke / e2e 可重复执行
- 开发服务器验证稳定
- 文档和验收标准齐全

## Phase 2 的定位

这一阶段不是生产化终点，而是从“最小闭环”升级为“可管理的控制平面”。

如果 Phase 2 成功，下一阶段再进入：
- 安全与可靠性增强
- 节点身份与认证
- 更严格的回滚机制
- 多协议设备接入

## 建议的执行节奏

1. 先冻结对象模型
2. 再做配置和部署参数化
3. 再补资产和任务编排
4. 再补审计和控制台
5. 最后补 smoke / e2e 和文档

## 备注

如果当前团队时间很紧，优先顺序应为：
1. 契约统一
2. 配置参数化
3. 资产管理
4. 任务编排
5. 审计记录
6. 控制台
7. 测试流程
8. 文档
