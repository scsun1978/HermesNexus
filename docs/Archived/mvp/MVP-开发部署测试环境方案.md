# HermesNexus MVP 开发 / 部署 / 测试环境方案

## 目标

建立一套最小但可重复的开发与验证环境，满足以下目标：

1. 本机负责快速开发与单元测试。
2. 开发服务器负责云端服务、边缘节点服务和联调验证。
3. Hermes 相关能力可用，但与当前主 Hermes 环境完全隔离。
4. MVP 阶段就能跑通“部署 -> 注册 -> 下发任务 -> 边缘执行 -> 结果回传 -> 云端可见”的最小闭环。

## 当前结论

- 适合本机开发。
- 不适合复用当前主 Hermes 环境。
- MVP 阶段必须同时建立开发服务器上的部署测试环境。
- 先做最小隔离和最小闭环，不要一开始就上复杂多 profile、HA、K8s、跨协议扩展。

## 推荐环境分工

### 1. 本机开发环境

用途：
- 写代码
- 跑单元测试
- 跑局部集成测试
- 调试 HermesNexus 本地逻辑

要求：
- 独立虚拟环境
- 独立 HERMES_HOME
- 独立 .env
- 独立 config.yaml
- 不污染当前主 Hermes 的 memory / sessions / skills / cache

建议：
```bash
cd /Users/shengchun.sun/Library/CloudStorage/OneDrive-个人/MyCloud/Code/HermesNexus
source .venv/bin/activate
export HERMES_HOME="$PWD/.hermes"
```

### 2. 开发服务器环境

用途：
- 部署云端服务
- 部署边缘节点服务
- 做真实联调
- 验证网络、端口、权限、SSH、回包路径

当前约定：
- 开发/测试服务器：scsun@172.16.100.101:22

要求：
- 能 SSH 登录
- 能部署 Docker / Docker Compose 或等价运行方式
- 能清晰区分云端服务与边缘节点服务
- 能重复部署和清理

## 最小目录与配置隔离

建议项目内至少具备：

```text
HermesNexus/
├── .hermes/
│   ├── config.yaml
│   ├── .env
│   ├── memory/
│   ├── skills/
│   ├── sessions/
│   ├── cache/
│   └── logs/
├── .venv/
├── cloud/
├── edge/
├── shared/
├── console/
├── deploy/
├── tests/
└── docs/
```

说明：
- .hermes 只给 HermesNexus 用
- 不要硬编码 ~/.hermes
- 不要把主 Hermes 的状态目录指向项目

## MVP 最小技术栈建议

保持简单，优先保证可部署和可验证：

- 云端：Python + FastAPI
- 边缘：Python 运行时
- 协议层：共享 schema / API contract
- 数据：先允许 MVP 级别的轻量存储，后续再升级持久化
- 部署：Docker Compose 起步

如果当前代码还没有完全稳定，不要急着扩展到更多协议或设备类型。

## MVP 部署拓扑

### 本机

- 代码编辑
- 单元测试
- 局部模拟

### 开发服务器

- cloud 服务
- edge 服务
- 测试数据库 / 缓存（如需要）
- 真实 SSH / 网络联调

建议把服务端和边缘节点都放在开发服务器上，先跑通“单机多进程/多容器”的最小闭环，再考虑分布式扩展。

## 最小闭环验证顺序

1. 启动 cloud 服务
2. 启动 edge 服务
3. edge 向 cloud 注册
4. cloud 创建任务
5. edge 拉取或接收任务
6. edge 通过 SSH 执行
7. 执行结果回传 cloud
8. cloud 可查询到任务与事件

## 测试层次

### 本机优先

- 单元测试
- 轻量集成测试
- 契约测试

### 开发服务器优先

- 部署验证
- 云边联调
- 端到端 smoke test

验证重点：
- 状态是否一致
- 日志是否完整
- 失败是否可回滚
- 任务是否重复执行
- 网络断开后是否能恢复

## 当前进入 MVP 的前置门槛

在继续扩展功能前，建议先满足：

- 本机独立开发环境已建立
- 开发服务器可访问
- cloud / edge 的最小启动方式已明确
- 配置与密钥边界已明确
- 最小测试命令已明确
- 当前阻塞测试已修复或至少已收敛

## 当前已知风险

- 全量 pytest 目前未绿
- SSH 执行器与测试契约有偏差
- cloud API 存在重复路由 / 未定义变量风险
- 事件顺序契约尚未统一
- 当前代码还需要先做稳定化修复，再做扩展

## 推荐下一步

### 第一阶段：环境隔离
- 建立项目专用 .venv
- 建立项目专用 HERMES_HOME
- 建立项目专用 .env / config.yaml
- 建立本机启动脚本

### 第二阶段：开发服务器最小部署
- cloud / edge 最小部署方式
- SSH / 端口 / 权限检查
- 部署与清理脚本

### 第三阶段：MVP 闭环验证
- 注册节点
- 下发任务
- SSH 执行
- 结果回传
- 任务可查询

### 第四阶段：稳定化
- 修复现有测试阻塞
- 收敛 API 契约
- 再进入下一轮功能扩展

## 建议的执行原则

- 本机做快速迭代，开发服务器做真实验证。
- Hermes 只作为项目内依赖，不作为全局前提。
- 先跑通最小闭环，再谈扩展能力。
- 任何新增环境变量、路径或服务端口，都要写进文档和启动脚本。
