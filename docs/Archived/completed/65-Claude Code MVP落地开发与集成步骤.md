# 65 Claude Code MVP 落地开发与集成步骤

## 目标

基于 `09-MVP开发路线图`，把 Hermes 的第一版 MVP 变成一套可由 Claude Code 直接执行的落地开发与集成步骤。

## 文档分工

- `09-MVP开发路线图.md`：定义 MVP 目标、环境边界和任务拆分原则
- `65-Claude Code MVP落地开发与集成步骤.md`：定义执行顺序和里程碑
- `66-Claude Code 可直接粘贴执行的任务提示词.md`：定义给 Claude Code 的执行提示词
- `67-第一轮Claude Code任务拆分.md`：定义第一轮可执行任务块

## MVP 范围

只做 Phase 1 的最小闭环：
- 1 个云端控制台/控制平面最小版
- 1 个边缘节点
- 少量 Linux 主机
- SSH 执行
- 任务下发
- 任务回传
- 基本审计

## MVP 成功标准

- 节点能注册到云端
- 云端能下发任务
- 边缘能执行 SSH 类动作
- 执行结果能回传并展示
- 失败能记录并可追踪
- 最少一条完整链路可跑通

## 环境约束

- 本机只负责开发、调试和单测。
- 开发服务器负责部署、联调和集成验证。
- 本次 MVP 的开发部署测试服务器为：`scsun@172.16.100.101:22`。
- 边缘节点只运行 edge runtime，不承担开发工具链。
- Hermes 作为受控依赖使用，不直接依赖系统全局环境。

## Claude Code 开发方式

建议用“先骨架、后闭环、再扩展”的顺序推进：

1. 先把仓库、配置、启动方式搭起来。
2. 再打通云端到边缘的最小通信链路。
3. 再接入 SSH 执行和任务回传。
4. 最后补上审计、测试、模拟器。

## 步骤 1：冻结 MVP 边界

### 交付
- 明确只做 Phase 1
- 锁定设备类型：Linux 主机
- 锁定协议：SSH
- 锁定任务类型：巡检、重启、升级、回滚中的“巡检 + 基础执行”

### Claude Code 任务
- 阅读 `09-MVP开发路线图`
- 阅读 `41-实施里程碑与交付拆分`
- 阅读 `58-代码仓库落地版总览`
- 输出一页 MVP 范围清单

## 步骤 2：建立仓库骨架

### 交付
- monorepo 目录结构
- cloud / edge / shared / drivers / console / deploy / tests / docs
- `.env.example`
- `docker-compose.yaml`
- `README.md`

### Claude Code 任务
- 创建目录结构
- 放入共享协议骨架
- 放入最小启动说明
- 生成本地开发启动命令

### 验证
- `make init` 或等价命令可运行
- `docker compose up` 能拉起基础依赖

## 步骤 3：先做共享契约

### 交付
- Job schema
- Event schema
- Node heartbeat schema
- Device schema
- Error code 约定

### Claude Code 任务
- 在 `shared/` 中定义数据结构
- 输出 OpenAPI 或 JSON Schema 草案
- 先冻结字段名，不写业务实现

### 验证
- 云端和边缘都能引用同一套 schema
- 至少 1 个请求和 1 个事件样例能通过校验

## 步骤 4：实现边缘节点最小闭环

### 交付
- 边缘节点进程
- 本地状态存储
- SSH 执行器
- 任务消费循环
- 任务结果上报

### Claude Code 任务
- 先实现 node heartbeat
- 再实现 task pull / task accept
- 再实现 SSH command 执行
- 再实现 result report

### 验证
- 边缘节点启动后能注册
- 能接受一个任务
- 能在一台测试 Linux 主机上执行命令
- 结果能返回云端

## 步骤 5：实现云端最小控制面

### 交付
- 节点注册 API
- 任务创建 API
- 任务状态查询 API
- 事件查询 API
- 最简控制台页面

### Claude Code 任务
- 先实现 identity / asset / task / event 的最小 API
- 再实现任务创建和状态查询
- 再实现控制台最小展示

### 验证
- 云端能看到节点在线
- 云端能发任务给边缘
- 云端能展示任务结果与日志摘要

## 步骤 6：接入审计与可观测

### 交付
- 审计记录表
- 事件流记录
- 基础指标
- 基础日志

### Claude Code 任务
- 给每次任务创建、执行、失败、回传都打审计
- 给节点心跳、任务状态、执行结果打事件
- 给关键路径加 trace_id

### 验证
- 任何一次任务都能追踪到是谁发起、谁执行、结果如何

## 步骤 7：补模拟器与联调工具

### 交付
- 节点模拟器
- SSH 设备模拟器
- 故障模拟器
- 联调脚本

### Claude Code 任务
- 写模拟 Linux 主机的脚本或容器
- 写一个假边缘节点或可控 edge stub
- 写一条端到端联调脚本

### 验证
- 不依赖真实站点也能完整跑通链路

## 步骤 8：补测试与回归

### 交付
- 单元测试
- 集成测试
- E2E 测试
- 失败回滚测试

### Claude Code 任务
- 先写任务状态机测试
- 再写 API 契约测试
- 再写边缘执行测试
- 再写失败场景测试

### 验证
- 关键链路都有自动化测试
- 失败场景可复现

## 步骤 9：整理发布与验收

### 交付
- MVP 发布说明
- 验收清单
- 回滚说明
- 已知问题列表

### Claude Code 任务
- 汇总本次 MVP 的功能范围
- 写验收标准
- 写部署步骤
- 写回滚步骤

### 验证
- 能在一个新环境重复部署
- 能在失败后回到已知状态

## 推荐 Claude Code 执行顺序

1. 读需求与路线图
2. 搭仓库骨架
3. 定共享契约
4. 做边缘闭环
5. 做云端控制面
6. 加审计与观测
7. 加模拟器
8. 加测试
9. 做发布与验收

## 每一步的 Claude Code 输出要求

每一步都要产出：
- 改了哪些文件
- 运行了哪些命令
- 验证结果是什么
- 还有哪些未完成项

## MVP 的最低可交付定义

只要这条链路通了，就算 MVP 成功：

“云端创建任务 → 边缘节点接收 → SSH 执行 → 结果回传 → 云端可见”

## 备注

这个步骤表故意压缩了范围，不追求一开始支持全部设备与全部协议。先把最小闭环打通，再往 Phase 2、3、4 扩展。
