# HermesNexus 开发服务器部署方案

## 目的

这份文档只解决一件事：MVP 阶段如何把本机开发和开发服务器验证分开。

结论先说：

1. 本机做开发。
2. 开发服务器做集成验证。
3. 开发服务器上同时部署 Cloud 服务和 Edge 节点。
4. 本机上的 Hermes 不直接拿来给项目主环境共用，必须做项目隔离。

## 为什么仍然要本机开发

即使本机已经装了 Hermes，HermesNexus 仍然建议本机开发，原因是：

- 改代码、跑单元测试、看日志都更快
- 不会频繁占用开发服务器资源
- 开发服务器只承担验证和联调，环境更稳定
- 你的现有 Hermes 可以继续保留，不会被项目改坏

正确做法不是“复用主 Hermes 环境”，而是“复用 Hermes 能力，隔离项目环境”。

## 推荐环境分层

### 1. 本机开发环境

职责：

- 代码编写
- 单元测试
- 本地调试
- 生成变更和文档

要求：

- 独立虚拟环境
- 独立 HERMES_HOME
- 独立 .env
- 不影响你日常主 Hermes

### 2. 开发服务器环境

职责：

- Cloud API 服务
- Edge Node 服务
- 集成测试
- 端到端验证
- 部署回归检查

要求：

- 固定地址和端口
- 固定测试数据域
- 可重复部署
- 可快速重启和回滚

## MVP 推荐拓扑

开发服务器上部署：

- 1 个 Cloud 服务
- 1 个 Edge 节点进程作为默认验证节点
- 后续如果需要，再加第二个 Edge 节点做多节点验证

通信方式：

- 本机通过 SSH / Git 同步代码到开发服务器
- 本机启动开发工作流
- 开发服务器暴露 8080 端口给验证访问

## 本机和服务器各做什么

### 本机

- 写代码
- 跑 pytest
- 检查配置
- 改文档
- 用本机 Hermes 做辅助开发，但限定在项目隔离环境里

### 开发服务器

- 跑 Cloud API
- 跑 Edge Node
- 做联调验证
- 保存集成日志
- 作为 MVP 验收环境

## 推荐目录与隔离方式

本机项目建议至少有：

```text
HermesNexus/
├── .venv/
├── .hermes/
├── .env
├── cloud/
├── edge/
├── tests/
└── docs/
```

其中：

- .venv 用于 Python 依赖隔离
- .hermes 用于项目自己的 Hermes 状态隔离
- .env 用于项目变量隔离

## 基本启动顺序

### 本机开发时

1. 激活项目虚拟环境
2. 设置项目专用 HERMES_HOME
3. 运行单元测试
4. 修改代码
5. 再跑单元测试

### 开发服务器验证时

1. 同步代码到开发服务器
2. 启动 Cloud 服务
3. 启动 Edge 节点
4. 做健康检查
5. 做任务下发和结果回传验证

## MVP 阶段建议保留的验证项

至少要跑通这些流程：

- Cloud API 健康检查
- 节点注册和心跳
- 任务创建和状态查询
- Edge 拉取任务并回写结果
- 服务重启后的恢复验证
- 日志和审计记录可追踪

## 不建议现在做的事

MVP 阶段不建议一开始就上：

- 复杂的多环境自动发布
- 复杂的多 profile 自动切换
- 生产级高可用编排
- 过重的 CI/CD 分层

先把“本机开发 + 开发服务器验证”跑顺，再扩展。

## 一句话结论

HermesNexus 的正确姿势是：本机开发，开发服务器验证，Hermes 主环境隔离使用，MVP 先跑通云边闭环。
