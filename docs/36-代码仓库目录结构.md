# 36 代码仓库目录结构

## 目标

定义 Hermes 分布式边缘设备管理系统的代码仓库如何拆分，方便多人协作、模块化开发和持续部署。

## 推荐仓库划分

### 1. 云端控制平面仓库
职责：
- API 服务
- 资产与任务服务
- 权限与审计
- Web 控制台

### 2. 边缘节点仓库
职责：
- Hermes Edge Node 运行时
- 设备适配器
- 本地任务队列
- 站点记忆与缓存

### 3. 设备驱动仓库
职责：
- 主机协议驱动
- IoT 协议驱动
- 网络设备协议驱动
- 厂商插件

### 4. 共享协议仓库
职责：
- 消息模型
- Job DSL
- 事件格式
- SDK / 客户端

### 5. 文档与样例仓库
职责：
- 架构说明
- 接入示例
- 部署脚本
- 运维手册

## 单仓库还是多仓库

### 单仓库优点
- 初期更简单
- 共享类型定义容易
- 改动联动更直接

### 多仓库优点
- 模块边界更清晰
- 便于团队分工
- 发布节奏更灵活

## 建议

- MVP 阶段：单仓库或 monorepo
- 稳定后：按服务和驱动拆分
- 协议、SDK、模型定义尽量单独沉淀

## Monorepo 示例

```text
hermes-platform/
├── cloud/
├── edge/
├── drivers/
├── shared/
├── console/
├── docs/
└── deploy/
```

## 设计原则

- 公共协议优先共享
- 边缘和云端边界清晰
- 驱动模块不要污染核心业务层
- 每个仓库要有自己的测试和发布流程
