# 11 云边通信与消息模型

## 目标

定义云端控制平面与边缘 Hermes 节点之间如何稳定、可追踪、可恢复地通信。

## 通信原则

- 命令与事件分离
- 支持断网缓存
- 所有消息可签名
- 所有任务可重放
- 传输必须可审计

## 通信模式

### 1. 命令下发
云端下发高层 Job：
- 目标设备
- 约束条件
- 回滚要求
- 超时策略
- 审批信息

### 2. 事件回传
边缘回传执行过程：
- job_started
- step_started
- step_finished
- step_failed
- rollback_started
- rollback_finished
- job_finished

### 3. 心跳
边缘节点周期性上报：
- 在线状态
- 负载
- 可执行能力
- 队列堆积
- 最近错误

## 推荐消息结构

```json
{
  "message_id": "uuid",
  "type": "job.execute",
  "tenant_id": "t1",
  "region_id": "r1",
  "node_id": "edge-01",
  "timestamp": "2026-04-10T10:00:00Z",
  "payload": {
    "job_id": "job-123",
    "target": "device-456",
    "action": "restart_service",
    "params": {
      "service": "nginx"
    }
  },
  "signature": "..."
}
```

## 可靠性策略

- Outbox pattern：先落本地，再异步发送
- Ack / retry：消息必须可确认
- Idempotency key：避免重复执行
- Dead-letter queue：失败消息单独处理
- 断线重连后做状态补偿

## 连接建议

- 云边主通道：mTLS 长连接
- 辅助通道：Webhook / Pull API
- 大流量事件：消息队列 / 流系统

## 实施建议

- 早期 MVP 用 HTTP + 签名 token 就够
- 稳定后升级成 mTLS + 队列
- 再接入消息总线做大规模分发
