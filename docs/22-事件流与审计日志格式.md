# 22 事件流与审计日志格式

## 目标

让每次任务执行、每次设备动作、每次审批都有统一日志结构。

## 事件类型

- node.online
- node.offline
- job.submitted
- job.assigned
- job.started
- job.step.started
- job.step.finished
- job.step.failed
- job.rollback.started
- job.rollback.finished
- job.finished
- audit.approved
- audit.rejected

## 统一事件格式

```json
{
  "event_id": "uuid",
  "trace_id": "trace-uuid",
  "timestamp": "2026-04-10T10:00:00Z",
  "tenant_id": "t1",
  "region_id": "r1",
  "node_id": "edge-01",
  "device_id": "device-456",
  "event_type": "job.step.failed",
  "severity": "error",
  "payload": {
    "job_id": "job-123",
    "step": "restart_service",
    "error": "timeout"
  }
}
```

## 审计字段

- actor
- action
- target
- approval_state
- risk_level
- reason
- evidence

## 事件设计原则

1. 追加写，不覆盖。
2. 事件必须带 trace_id。
3. 错误必须包含上下文。
4. 审计必须能复原关键路径。
5. 事件字段要稳定，避免频繁改 schema。

## 观测建议

- 云端写入事件湖
- 边缘本地保留短期事件缓存
- 高价值事件单独归档
