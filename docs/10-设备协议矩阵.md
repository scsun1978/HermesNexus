# 10 设备协议矩阵

## 目标

把不同类型设备接入 Hermes 系统时的协议、认证方式、读写能力和风险等级统一起来。

## 协议分类

### 主机类

| 协议 | 用途 | 适合场景 | 风险 |
|---|---|---|---|
| SSH | 命令执行、文件传输、服务控制 | Linux / Unix 主机 | 中 |
| WinRM | Windows 远程管理 | Windows 主机 | 中 |
| Agent | 长连接、主动上报 | 关键主机、受管机 | 低-中 |

### IoT 类

| 协议 | 用途 | 适合场景 | 风险 |
|---|---|---|---|
| MQTT | 状态订阅、控制消息 | 传感器、网关、边缘终端 | 低 |
| HTTP/REST | 配置、遥测、控制 | 通用 IoT 设备 | 低 |
| CoAP | 轻量设备通信 | 资源受限设备 | 低 |
| Modbus | 工业控制 | 工控 / 传感器 / PLC | 高 |

### 网络设备类

| 协议 | 用途 | 适合场景 | 风险 |
|---|---|---|---|
| SSH | CLI 管理 | 路由器、交换机、防火墙 | 高 |
| NETCONF | 配置管理 | 支持模型驱动配置的设备 | 高 |
| RESTCONF | HTTP 配置接口 | 新一代网络设备 | 中-高 |
| SNMP | 状态采集、告警 | 监控与发现 | 低 |

## 统一能力视图

每个协议都要映射到统一能力：
- read_status
- push_config
- backup_config
- diff_config
- restart_service
- upgrade_firmware
- rollback
- subscribe_events

## 认证方式

- SSH key / 用户名密码 / jump host
- API token / OAuth / mTLS
- SNMP community / v3 auth
- 设备证书 / 短期凭证

## 设计原则

1. 适配器必须声明能力，不允许“伪支持”。
2. 高危协议需要更严格的授权与审计。
3. 所有设备操作尽量幂等。
4. 读取和写入必须分离建模。
5. 协议细节不能泄露到云端编排层。

## 建议的适配器元数据

```yaml
name: ssh-host
capabilities:
  - read_status
  - push_config
  - rollback
risk: medium
transport: ssh
```
