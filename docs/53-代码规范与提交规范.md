# 53 代码规范与提交规范

## 目标

让开发团队在同一套规范下协作，减少风格不一致和回归问题。

## 代码规范

### 1. 清晰优先
- 变量名要可读
- 函数职责单一
- 复杂逻辑要拆分

### 2. 结构优先
- 云端、边缘、驱动、共享协议边界明确
- 公共数据结构集中管理

### 3. 可测试优先
- 核心逻辑要能单元测试
- 外部依赖通过接口注入

### 4. 可审计优先
- 写操作必须有日志
- 高危动作必须留痕

## 提交规范

- feat: 新功能
- fix: 修复问题
- refactor: 重构
- docs: 文档
- test: 测试
- chore: 杂项

## 提交原则

- 每次提交只做一件事
- 提交信息必须能看懂
- 不要把格式化和业务改动混在一起
- 关键修复要带验证说明

## PR 建议

- 说明改了什么
- 说明为什么改
- 说明怎么验证
- 说明有没有风险

## Hermes 开发风格

- 业务逻辑优先结构化
- 协议和适配器解耦
- 所有边界都要可追踪
