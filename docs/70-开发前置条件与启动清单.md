# 70 开发前置条件与启动清单

## 目标

在正式进入 HermesNexus 的 MVP 开发前，把仓库、环境、配置、部署、测试和排障的最小前置条件一次性补齐，避免开发过程中反复返工。

## 范围

这份清单只覆盖“开始开发前必须确认的事情”，不展开具体实现细节。

## 一、仓库工程骨架

### 必须确认

- [ ] 仓库目录结构已确定
- [ ] cloud / edge / shared / console / deploy / tests / docs 这些主目录已就位
- [ ] Python 虚拟环境方案已确定
- [ ] 前端项目初始化方案已确定
- [ ] Makefile / scripts 入口已确定

### 结果标准

- 能看出哪些目录负责云端、边缘、共享协议、前端、部署和测试
- 新人可以快速理解仓库边界

## 二、本机开发环境

### 必须确认

- [ ] 本机使用独立虚拟环境
- [ ] HermesNexus 与当前主 Hermes 环境隔离
- [ ] 已明确 `HERMES_HOME` 的项目级路径
- [ ] 已有项目专用 `.env`
- [ ] 已有项目专用 `config.yaml` 模板

### 结果标准

- 本机开发不会污染现有 Hermes 的配置、记忆和会话
- 项目命令能在独立环境中运行

## 三、开发部署测试服务器

### 必须确认

- [ ] 开发部署测试服务器已明确：`scsun@172.16.100.101:22`
- [ ] SSH 可达
- [ ] 部署账号权限足够
- [ ] Docker / Docker Compose 可用
- [ ] 必要端口已规划
- [ ] 防火墙 / 安全组策略已确认

### 结果标准

- 可以在服务器上部署服务端和边缘节点
- 可以在服务器上做集成测试和回归验证

## 四、最小技术栈

### 必须确认

- [ ] 云端后端技术栈已定
- [ ] 边缘运行时技术栈已定
- [ ] 共享协议层技术栈已定
- [ ] 控制台前端技术栈已定
- [ ] 部署方式已定为 Docker Compose 起步

### 推荐基线

- 云端 / 边缘：Python 3.12 + FastAPI + Pydantic
- 数据：PostgreSQL + Redis + SQLite
- 前端：Next.js + TypeScript + Tailwind CSS + shadcn/ui
- 部署：Docker Compose
- 记忆：MVP 先抽象 provider，OpenViking 延后

## 五、配置与密钥

### 必须确认

- [ ] `.env.example` 已有
- [ ] `config.example.yaml` 已有
- [ ] secrets 不入 git
- [ ] 证书、Token、SSH key 的管理方式已明确
- [ ] 云端 / 边缘 / 本机的配置边界已明确

### 结果标准

- 配置可以复制、替换、部署
- 不会因为配置混杂导致环境不可重复

## 六、最小数据模型

### 必须确认

- [ ] Node 模型
- [ ] Job 模型
- [ ] Event 模型
- [ ] Device 模型
- [ ] Audit 模型
- [ ] Memory provider 接口

### 结果标准

- MVP 核心链路的字段定义清晰
- 云端、边缘、控制台对数据模型的理解一致

## 七、运行和验证方式

### 必须确认

- [ ] 一键启动命令已定义
- [ ] 一键停止命令已定义
- [ ] 一键测试命令已定义
- [ ] 一键清理命令已定义
- [ ] 最小 smoke test 已定义

### 结果标准

- 开发者知道如何启动、验证和回滚
- 每次改动都能快速检查是否破坏最小闭环

## 八、质量基线

### 必须确认

- [ ] 格式化工具已确定
- [ ] lint 工具已确定
- [ ] 单元测试框架已确定
- [ ] 集成测试框架已确定
- [ ] 最低验收标准已确定

### 结果标准

- 新代码进入仓库前有基本质量门槛
- 关键链路有可重复验证手段

## 九、观测与排障

### 必须确认

- [ ] 结构化日志格式已确定
- [ ] trace_id / request_id 策略已确定
- [ ] health endpoint 已确定
- [ ] 基础故障排查流程已确定

### 结果标准

- 出问题时能快速定位到节点、任务和请求
- 不是只能靠人工猜测

## 十、MVP 启动门槛

只有满足下面这些条件，才建议进入正式开发：

- [ ] 仓库骨架明确
- [ ] 本机开发隔离明确
- [ ] 开发部署测试服务器可用
- [ ] 技术栈明确
- [ ] 配置和密钥边界明确
- [ ] 最小数据模型明确
- [ ] 启动和测试命令明确
- [ ] 质量基线明确
- [ ] 观测和排障路径明确

## 建议执行顺序

1. 先定仓库骨架
2. 再定本机隔离
3. 再定开发服务器
4. 再定技术栈
5. 再定配置和密钥
6. 再定数据模型
7. 再定命令和测试
8. 最后再开始写业务代码

## 备注

- 如果前置条件不完整，后面的开发会很容易返工。
- 这份清单的目的不是拖慢开发，而是把 MVP 的启动成本降到最低。
